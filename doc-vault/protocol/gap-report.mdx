This report compares the Knify Canvas Protocol against OpenAI's Symphony SPEC (`openai/symphony` SPEC.md) on the dimensions that determine whether a third party can implement the protocol from the spec alone, without reading the reference implementation.

Symphony and Knify solve different problems — Symphony is a coding‑agent daemon, Knify is a content protocol for typed‑block canvases — so we do not compare them on domain content. We compare them on the **shape of a spec that outsiders can implement to**: normative language, state machines, error model, conformance criteria.

The protocol exposes three implementer roles:

- **Agent (publisher)** — produces canvases and consumes feedback callbacks
- **Hub (server)** — stores canvases, fans out events, delivers feedback
- **Viewer / API consumer** — reads canvases, exports, and event streams

The Knify protocol MUST be implementable independently by each of these roles.

## Summary

The protocol files at [doc-vault/protocol/](.) cover the canvas object shape, the patch‑event idea, and a sketch of an exports API. The wire contract that a real implementer needs — auth, errors, state machines, the live‑events transport, the feedback‑callback shape, the full route table, the block taxonomy — currently lives implicitly in [apps/hub/src/types.ts](../../apps/hub/src/types.ts) and [apps/hub/src/app.ts](../../apps/hub/src/app.ts).

This report identifies 18 structural gaps. Eleven new spec files and four edits to existing files close them. See [Files added or modified](#files-added-or-modified).
